Architecture Approaches

Three ways to build the Claude session multiplexer + observatory

A

PTY Multiplexer (Recommended)

Endeavor owns the terminal. Uses node-pty to spawn/manage Claude Code processes in pseudo-terminals. TUI renders via Ink (React for terminals). Discovers existing sessions via Claude Code's ~/.claude/ state files and offers to "adopt" them by reading their output streams.

Pros

  • Full control over I/O — can capture, replay, inject input
  • Real-time token/cost tracking via output parsing
  • Ink gives component model + hot reloading for TUI dev
  • node-pty is battle-tested (powers VS Code terminal)

Cons

  • Attaching to existing sessions is hard — can only read state, not embed their terminal
  • node-pty is a native module (compilation on install)
  • Must handle SIGWINCH, color codes, cursor positioning
B

tmux Backend

Use tmux as the session multiplexer under the hood. Each Claude session runs in a tmux pane. TUI is a custom tmux interface (or a separate process that reads tmux capture-pane). Existing sessions can be moved into tmux windows.

Pros

  • tmux handles all the hard PTY problems (attach/detach, resize, persistence)
  • Existing sessions can be genuinely attached via tmux move-pane
  • Sessions survive Endeavor crashes
  • Mature, stable foundation

Cons

  • Hard dependency on tmux being installed
  • TUI is constrained by tmux's own rendering model
  • Custom dashboard UI requires fighting tmux's layout system
  • Parsing tmux capture-pane for status extraction is fragile
C

Hybrid: Observer + Launcher

Split the problem. The observer reads Claude Code's state files, hooks, and process list to build the dashboard — no PTY needed. The launcher uses node-pty only for sessions Endeavor spawns. Existing sessions stay in their original terminals but are observed remotely.

Pros

  • Observer works for ALL session types without PTY complexity
  • Existing sessions don't need to be "adopted" — just watched
  • Clean separation: observe everything, control what you spawn
  • Easiest path to adding API/cloud session adapters later

Cons

  • Can't type into existing sessions from the TUI — observe only for those
  • Two different UX models (full control vs. read-only) depending on session origin
  • Observer data is only as good as Claude Code's state files expose
